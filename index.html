01 What is Fable, really?

Imagine the smartest, most patient helper you've ever met.

Before this helper says anything, it goes very quiet and thinks really hard — sometimes for a long time, like a chess player staring at the board. You can't watch it think; it does that part in its head. Then it gives you a careful answer.

It's so patient it can work on one big job for many minutes without stopping — reading lots of things, trying stuff, checking its own work — instead of just blurting out the first idea.

It's brilliant, but it's a new kind of helper, so a few of the old buttons and tricks you used with its older cousins don't work the same way anymore. That's what this guide is about: which buttons changed, and how to talk to it so it does its best work.

👉 Flip the switch at the top to "a developer" when you want the real, technical version.

Fable 5 is a frontier‑tier model with a distinct API contract.

It shares the Messages API shape with the Opus family, but it is not a drop‑in for Opus — it changes how thinking is configured, how reasoning is returned, how tokens are counted, how long a single turn runs, and how refusals surface. Treat a migration to Fable as a behavior change, not just a model‑string swap.

  • Reasoning is always‑on and protected. Thinking happens on every request; the raw chain of thought is never returned to you (summaries only).
  • It's tuned for long‑horizon autonomy. Single requests on hard tasks can legitimately run many minutes at higher effort.
  • 1M context at standard pricing, 128K max output, new tokenizer (~30% more tokens than Opus‑tier for the same text).
  • Same call site as Opus — client.messages.create(model="claude-fable-5", …) — minus the parameters listed in §02 and §03.

Use Fable only when you've chosen it. For a generic "use the latest model," the default upgrade target is still claude-opus-4-8 — Fable costs more and re‑baselines your token math.

02 The seven things that are different

Each card is a real, behavioral difference from the Opus/Sonnet models. These are the source of nearly every "wait, why isn't this working" moment. Tap one to expand.

🧠

Thinking is always on

There is no "off." Don't send a thinking parameter at all.

On Opus you could disable thinking or set a budget_tokens. On Fable both error with a 400. Thinking applies automatically. You steer how hard it thinks with output_config.effort — low · medium · high · xhigh · max — not a token budget.

🔒

You never see the raw reasoning

"Protected thinking." Best you get is a summary.

Responses carry thinking blocks, but the raw chain of thought is never exposed. display:"summarized" gives a readable summary; the default ("omitted") leaves the text empty. When you continue a conversation on Fable, echo those blocks back exactly as received — editing them is a 400.

🔢

A new tokenizer

Same text ≈ 30% more tokens than Opus‑tier.

Old token counts, context budgets, and max_tokens values don't transfer. Re‑measure with count_tokens (passing model:"claude-fable-5" returns counts under both tokenizers so you can see the delta). Billing is per token, so an unchanged workload can simply cost more.

⏱️

Turns can take minutes

A 15‑minute single request is normal on hard tasks.

It gathers context, builds, and verifies before answering. Plan for it: stream, set generous timeouts, show progress, and structure work so callers check back asynchronously instead of blocking on one synchronous request.

🛑

Refusals come back as a success

HTTP 200 with stop_reason:"refusal".

Safety classifiers (mainly bio + most cyber) can decline a request. It's a normal 200, so code that reads response.content[0] blindly will crash on an empty array. Check stop_reason first. You can auto‑retry on another model with the fallbacks parameter.

✍️

No assistant prefill

Can't seed the answer with a leading assistant turn.

Ending your messages on an assistant turn to force a shape returns a 400. Use structured outputs (output_config.format with a JSON schema) or a system‑prompt instruction instead. Same restriction as the 4.6+ family.

📦

30‑day retention required

Not available under zero‑data‑retention.

If your org is set to ZDR (or any retention below 30 days), every Fable request returns a 400 invalid_request_error — even a perfectly valid payload. If a migration suddenly 400s for no obvious reason, check the org's retention setting before debugging the body.

03 Pitfalls & gotchas people miss

The differences above turn into bugs in predictable ways. Here's each trap with the exact fix. Click to open; hit Copy on any snippet.

#1 Sending a thinking or budget_tokens parameter

❌ The trap: You copy an Opus snippet that sets thinking={"type":"enabled","budget_tokens":10000} or thinking={"type":"disabled"}. Both return 400 on Fable.

✅ The fix: Omit thinking entirely (adaptive applies automatically). Control depth with output_config.effort.

# ❌ wrong — both of these 400 on Fable
client.messages.create(model="claude-fable-5", max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 10000}, messages=msgs)
client.messages.create(model="claude-fable-5", max_tokens=16000,
    thinking={"type": "disabled"}, messages=msgs)
# ✅ right — no thinking param; steer with effort
client.messages.create(
    model="claude-fable-5",
    max_tokens=16000,
    output_config={"effort": "high"},   # low | medium | high | xhigh | max
    messages=msgs,
)
#2 Reading response.content[0] without checking for a refusal

❌ The trap: A safety classifier declines the request. You still get HTTP 200, but content is empty (pre‑output) or partial (mid‑stream). Indexing into it throws.

✅ The fix: Branch on stop_reason before touching content. (Branch on stop_reason, never on stop_details — that can be null even on a refusal.)

resp = client.messages.create(model="claude-fable-5", max_tokens=1024, messages=msgs)

if resp.stop_reason == "refusal":
    # pre-output: content is empty and you were NOT billed
    # mid-stream: partial output WAS billed — discard it
    handle_refusal(resp.stop_details)   # .category: "cyber" | "bio" | None
else:
    print(resp.content[0].text)
#3 Reusing your old max_tokens / cost math

❌ The trap: The new tokenizer produces ~30% more tokens for the same content. Budgets, compaction triggers, and cost estimates calibrated on Opus are now wrong — and you can silently truncate output mid‑thought.

✅ The fix: Re‑baseline with count_tokens. Passing model:"claude-fable-5" returns counts under both tokenizers so you can see the delta on your prompts. Never apply a blanket multiplier.

r = client.messages.count_tokens(model="claude-fable-5", messages=msgs)
print(r.input_tokens)                  # new tokenizer — what you're billed
print(r.input_tokens_prior_tokenizer)  # same request under the old tokenizer
#4 A non‑streaming request with a big max_tokens

❌ The trap: Fable can run for minutes and emit up to 128K tokens. A blocking, non‑streaming call hits SDK/HTTP timeouts and the connection drops — the SDK will even refuse the request up front.

✅ The fix: Stream anything large or long. Use the SDK helper to get the assembled message when you don't need per‑token events.

with client.messages.stream(
    model="claude-fable-5",
    max_tokens=64000,
    output_config={"effort": "high"},
    messages=msgs,
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
    final = stream.get_final_message()   # full Message, with usage + stop_reason
#5 Stripping or editing thinking blocks before sending them back

❌ The trap: In a multi‑turn or tool loop you trim the assistant turn down to just the text, or you "clean up" the thinking blocks. On Fable, replaying modified blocks is rejected.

✅ The fix: Append the full response.content back, unchanged — including empty‑text thinking blocks. (Continuing on a different model? It silently drops Fable's protected blocks for you — no stripping needed, and they aren't billed.)

# continuing the SAME conversation on Fable:
messages.append({"role": "assistant", "content": resp.content})  # verbatim
messages.append({"role": "user", "content": tool_results})
#6 Wanting to read the model's reasoning and getting blank blocks

❌ The trap: You stream to a UI and see a long pause, then output — because the default display is "omitted" (thinking blocks arrive with empty text). It looks frozen.

✅ The fix: Ask for the summary explicitly. (You still never get the raw chain of thought — and don't try to prompt for it; that can be refused with category:"reasoning_extraction".)

client.messages.create(
    model="claude-fable-5",
    max_tokens=16000,
    thinking={"type": "adaptive", "display": "summarized"},  # readable progress
    messages=msgs,
)
#7 Forcing output shape with an assistant prefill

❌ The trap: The classic {"role":"assistant","content":"{\""} prefill to force JSON. Returns a 400 on Fable.

✅ The fix: Use structured outputs — guaranteed schema‑valid, no prefill.

resp = client.messages.create(
    model="claude-fable-5",
    max_tokens=16000,
    output_config={"format": {"type": "json_schema", "schema": MY_SCHEMA}},
    messages=[{"role": "user", "content": "Extract the contact details."}],
)
# in the SDK you can also use client.messages.parse(...) with a Pydantic/Zod model
#8 Leaving aggressive, step‑by‑step prompts in place

❌ The trap: Prompts written for older models — "CRITICAL: YOU MUST…", numbered step lists, "double‑check the layout before returning" — actually reduce Fable's output quality. It follows them literally and over‑plans or over‑tidies.

✅ The fix: State the goal and the constraints; remove the scaffolding. A/B your workload with the old step‑by‑step instructions deleted. See §04 Prompting for drop‑in snippets.

#9 Every request 400s and the payload looks fine

❌ The trap: You've triple‑checked the request body and still get 400 invalid_request_error on everything.

✅ The fix: Check the org's data‑retention setting. Fable requires 30‑day retention and is unavailable under ZDR — that's an org‑level 400, not a payload problem. Also confirm you removed temperature / top_p / top_k (all removed — they 400 too).

04 How to talk to Fable

Don't over‑explain. Give it the goal, tell it the rules, then get out of the way.

Older helpers needed you to say "STEP 1, STEP 2, and PLEASE don't forget…". Fable is smart enough that all that nagging actually makes it worse — like a chef who gets distracted if you stand over their shoulder narrating. Tell it what "done" looks like and let it cook.

The little snippets below are polite, plain‑English rules you can paste into the "system" instructions to keep it tidy, honest, and not over‑doing things. Hit Copy and use them.

Fable is highly steerable but under‑prompted is usually better than over‑prompted.

The big knobs:

  • output_config.effort is your primary control. Default to high; use xhigh for the most capability‑sensitive work; drop to low/medium for routine tasks. Notably, low on Fable often beats xhigh/max on prior models — sweep it.
  • Give the full task spec up front in one well‑specified turn for long‑horizon work. Ambiguous, progressively‑revealed prompts reduce both efficiency and quality.
  • Give the why, not just the request. Fable connects the task to context better when it knows the intent.
  • Invest in a communication‑style section — Fable responds strongly to it, and un‑steered it can over‑elaborate (heavy PR descriptions, unrequested refactors, narrated comments).

The snippets below are battle‑tested system‑prompt addenda for exactly these behaviors.

Don't over‑plan

for ambiguous tasks
When you have enough information to act, act. Do not re-derive facts
already established, re-litigate a decision the user has made, or
narrate options you won't pursue. If you're weighing a choice, give a
recommendation, not an exhaustive survey. (This does not apply to
thinking blocks.)

Don't tidy / over‑engineer

at higher effort
Don't add features, refactor, or introduce abstractions beyond what the
task requires. A bug fix doesn't need surrounding cleanup. Don't design
for hypothetical future requirements — do the simplest thing that works.
Don't add error handling or validation for scenarios that cannot happen;
only validate at system boundaries (user input, external APIs).

Ground progress claims

long agentic runs
Before reporting progress, audit each claim against a tool result from
this session. Only report work you can point to evidence for; if
something isn't verified, say so. Report outcomes faithfully: if tests
fail, say so with the output; if a step was skipped, say that; when
something is done and verified, state it plainly without hedging.

Respect boundaries

avoid unrequested actions
When the user is describing a problem, asking a question, or thinking
out loud rather than requesting a change, the deliverable is your
assessment. Report your findings and stop. Don't apply a fix until they
ask. Before running a state-changing command (restarts, deletes, config
edits), check the evidence actually supports that specific action.

Delegate to sub‑agents

parallelizable work
Delegate independent subtasks to sub-agents and keep working while they
run. Prefer sub-agents that communicate asynchronously over
spawn-and-block. Intervene if a sub-agent goes off track or is missing
relevant context.

Use a memory file

recurring work
Store one lesson per file with a one-line summary at the top. Record
corrections and confirmed approaches alike, including why they mattered.
Don't save what the repo or chat history already records; update an
existing note rather than duplicating; delete notes that turn out wrong.

Stay autonomous

unattended / async pipelines
You are operating autonomously. The user is not watching and cannot
answer mid-task, so asking "Want me to…?" will block the work. For
reversible actions that follow from the request, proceed without asking.
Before ending your turn, check your last paragraph: if it's a plan, a
question, or a promise about work you haven't done, do that work now.

Write readable summaries

long sessions
When you write the final summary, drop the working shorthand. Open with
the outcome: one sentence on what happened or what you found. Then the
supporting detail. Use complete sentences; spell terms out; no arrow
chains or made-up labels the reader hasn't seen. If you must choose
between short and clear, choose clear.
